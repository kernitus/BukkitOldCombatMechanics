# Total downloads README badge plan

## Goal

Add a README badge that shows an aggregate OldCombatMechanics download count across GitHub Releases, Spigot, BukkitDev/CurseForge, and Hangar. The badge should be generated by Shields.io from JSON published by GitHub Actions to GitHub Pages, avoiding routine commits to the main branch.

## Locked decisions

- Badge display route: Shields.io endpoint badge.
- Stats generation route: scheduled GitHub Actions workflow.
- Publication route: GitHub Pages, not commits to `main`.
- Preferred cadence: daily, with a manual `workflow_dispatch` trigger for ad-hoc refreshes.
- Main branch should not receive recurring download-count commits.
- The total may be cached rather than truly live; daily freshness is acceptable.
- Using repository secrets for platform API access is acceptable where needed; the repository already has release-process API keys, so a `CURSEFORGE_API_KEY`-style secret is in scope if the chosen BukkitDev/CurseForge endpoint requires it.

## Proposed implementation

### Slice 1: Add the stats generator

Intended outcome: a repository script can collect per-platform counts, combine them, and emit Shields-compatible badge JSON plus detailed machine-readable stats.

- Add a small script under a suitable path such as `scripts/download_stats.py` or `.github/scripts/download_stats.py`.
- Fetch GitHub release asset download counts through the GitHub REST API using the workflow `GITHUB_TOKEN`.
- Fetch Hangar downloads using Hangar's public API for the configured project slug.
- Fetch BukkitDev/CurseForge downloads using the most stable available API path; prefer an official API over scraping and read any API key from repository secrets.
- Fetch Spigot downloads using the least brittle available source; if no official API is available, isolate scraping in a single function with conservative request headers and clear failure reporting.
- Before writing the final generator logic for any non-GitHub platform, document the chosen endpoint/source in code comments or a small source map so future maintainers can see what is API-backed versus scraped.
- Produce at least two files in an output directory:
  - `downloads.json`: Shields endpoint JSON, for example `{ "schemaVersion": 1, "label": "downloads", "message": "1.2M", "color": "blue" }`.
  - `download-stats.json`: detailed source values, total, update time, and per-source status/warnings.
- Format totals compactly for badge display, e.g. `999`, `1.2k`, `123k`, `1.2M`.
- Avoid Java/Kotlin project changes unless needed; this is repository automation, not plugin runtime code.

Verification:

- Run the script locally with test identifiers or mocked responses where practical.
- Validate both JSON files parse successfully.
- Confirm partial failures are represented in `download-stats.json` without producing misleading totals.

### Slice 2: Add the GitHub Actions Pages workflow

Intended outcome: GitHub Actions can generate and publish the JSON files to Pages on a schedule without committing generated stats to `main`.

- Add `.github/workflows/update-download-stats.yml`.
- Triggers:
  - `schedule` daily.
  - `workflow_dispatch`.
- Permissions:
  - `contents: read`.
  - `pages: write`.
  - `id-token: write`.
- Steps:
  - Check out the repository.
  - Set up the scripting runtime if needed.
  - Run the stats generator into a public output directory.
  - Configure Pages using `actions/configure-pages`.
  - Upload the directory as a Pages artefact.
  - Deploy to GitHub Pages using `actions/deploy-pages`, with the standard `github-pages` environment.
- Add a short timeout, for example 5 minutes.
- Do not commit generated JSON back to the repository.
- Implement previous-stats reuse if feasible by fetching the currently published `download-stats.json` from Pages before generation; if that fetch is unavailable on first run, proceed with clear per-source status reporting.

Verification:

- Validate workflow YAML syntax structurally.
- Pre-merge review must confirm the workflow includes the required Pages permissions, `actions/configure-pages`, `actions/upload-pages-artifact`, `actions/deploy-pages`, the `github-pages` deployment environment, and both scheduled/manual triggers.
- Run the generator step locally as far as credentials allow.
- After merge, run `workflow_dispatch` and confirm Pages serves both JSON files. This requires the repository's Pages source to be set to GitHub Actions.

### Slice 3: Add the README badge

Intended outcome: the README shows a stable aggregate downloads badge backed by the GitHub Pages JSON endpoint.

- Add a README badge using Shields endpoint syntax, tentatively:
  - `![Downloads](https://img.shields.io/endpoint?url=https://kernitus.github.io/BukkitOldCombatMechanics/downloads.json)`
- Place the badge below the current centred project header/images and above the main introductory text, as a small centred badge row, unless the README already has a conventional badge block by implementation time.
- Optionally link the badge to `download-stats.json` or the project's download page if preferred.
- Document, briefly, that the count is updated daily and aggregates GitHub, Spigot, BukkitDev/CurseForge, and Hangar.

Verification:

- Once Pages has deployed, confirm Shields can read the endpoint JSON.

## Failure and data-quality policy

- Prefer official APIs over scraping.
- Treat Spigot scraping as brittle and isolated; do not let scraper details leak into the workflow.
- If one source fails during a scheduled run, prefer keeping the previous known value for that source if a previous stats file is available; otherwise report the failed source clearly and exclude it from the computed total rather than inventing a value.
- Record per-source `status`, `value`, and `warning` fields in the detailed JSON so future triage is possible.
- Do not fail the whole workflow for a single transient platform failure unless all sources fail or the output JSON would be invalid.
